# 大型建筑协作提示词模板

将尖括号内容替换后交给 AI。第一次使用先通过 scaffold 创建 work。

```text
请按照 knowledge/promot/large_structure_workflow/SKILL.md 完成
workset/<work_name> 中的大型建筑。

需求：
<建筑主题、世界观、用途、规模感>

必须出现：
- <地标 1>
- <区域 2>
- <玩法或叙事空间 3>

玩家体验：
- 玩家从负 Z 的推荐传送点接近。
- 主路线为：<入口 → 前厅 → ... → 核心/Boss 区>。
- 必须可进入的内部：<房间列表>。
- 希望第一眼看到：<轮廓与视觉焦点>。

内部空间与照明：
- 大型建筑及楼层：<建筑 -> 楼层列表>。
- 每栋大型建筑的房间/功能区：<建筑 -> 门厅、核心空间、附属房、后场、楼梯等>。
- 内部路线：<主入口 -> 各房间/楼层 -> 出口>。
- 内饰识别锚点：<御座、柜台、灶台、档案架、床组、供桌等>。
- 实体分间：每个 RoomPlan 声明 enclosed/screened/open_zone；临时封住门洞后对室内空气 flood fill，实际分区不得少于计划分区。家具、地毯、书架和房间名称不能冒充墙、屏风与门洞。
- 门与路线优先：先声明 PortalPlan 和从室外入口到各房间/楼梯的 RoutePlan，冻结门洞、门内外 3×3 缓冲与主通道保护体积，再生成带开口的隔墙；禁止实墙覆盖门洞后再零散 clear。
- 封闭隔墙逐列从地板连接到水平天花、梁底或坡屋顶下表面；门洞上方恢复门楣和墙体。两格高矮墙只能用于 screened/open_zone，不能算 enclosed 房间。
- 从室外入口前点对最终方块做 BFS，必须穿过门槛和门内缓冲，到达全部公共房间、楼梯与出口；入口后撞墙、侧挤一格缝或被家具截断均失败。
- 大型住宅、客栈、衙署等每层至少 2 个真实空气分区，优先 3 个；除有意大厅外，单一无隔断空间不超过该层可用面积的 60%。
- 家具按房间用途组成多方块组团：每个房间至少一个主锚点和一个辅助组，大型核心房间至少 3 个组团；禁止用零散书架、工作台和木桶填满所有房间。
- 每个房间登记入口 viewpoint、主锚点、至少两个辅助证据和 activity_zone；从门口应能看见/抵达主锚点，家具要有使用侧。隐藏房间名称后仍应能从入口截图大致判断用途。
- 家具模板按建筑面积至少使用 3/5/7 种（160–255 / 256–511 / 512+），相邻大型房屋不能复用完全相同的内部坐标布局。
- 夜间照明：每个房间、走廊、楼梯平台和入口都要覆盖；所有光源统一登记并审计暗点。
- 屋面围护：除俯视投影外，检查山墙、侧坡、墙顶接缝和外部空气漏入室内。

结构支撑与全路线：
- 大型上部体量：<城门楼、角楼、塔楼、大殿等>；逐项说明台基/城台、柱墙和地面的荷载路径。
- 允许的悬挑：<只允许屋檐挑出；屋身和楼板不得悬空>。
- 玩家全路线：<地面入口 -> 楼梯/马道 -> 上层/巡道 -> 门楼/角楼 -> 返回出口>。
- 对所有声明的可行走格做 BFS；少量采样点、入口存在或房间名称不能代替真实连通。
- 对屋檐、瓦垄、搏风板、脊兽、斗拱等细部做六方向连通审计，禁止对角接触和孤立方块。

细部方块状态与园林：
- 悬挂组件：<灯具/牌匾等>；铁链必须自实体梁架连续连接到悬挂物，顶端上方不得是空气，灯具悬挂状态必须正确。
- 可行走楼梯：<位置与低/高平台>；按实际上升向量生成并审计 Bedrock 方向状态、净空和落点。
- 楼梯井：在楼板、墙、梁和家具生成前登记 reserved_stair_volume；检查玩家约 0.6×1.8 格碰撞体沿每一级踏步扫过的空间，而不是只检查踏步正上方一格。
- 大型住宅和公共建筑楼梯优先至少 2 格宽；每阶上方连续两格可通过，转向平台、最上一级和楼板洞口优先三格高；楼板洞口沿上升方向额外留 1 格。
- 屋面/装饰楼梯：<屋面、座椅、斗拱等>；旋转和镜像时同时变换坐标与方块状态。
- 种植分区：<区域 -> 风格、主种、辅种、允许基底、禁入路线>；花草下方必须是该分区声明的可种植基底。
- 花草按有边界的组团与留白构图，禁止随机散点、棋盘格、机械等距序列和每格换一种花。
- 景观不得占用门前 3×3、楼梯平台、主路、巡道或两格宽通道，也不得穿入屋顶和立面。

超大型都市（画布任一水平轴不少于 1024 时必填）：
- 逻辑扇区：<扇区大小与数量>；先冻结跨区礼仪轴、市井路网、水运网、护城河、城墙和接缝保护带。
- 城区清单：<皇城、官署、军营、豪华院落、商业、民居、手工业、漕运仓储、文化宗教、滨水娱乐、城外服务等>。
- 建筑唯一性：除<明确豁免的皇城/兵营秩序性组团>外，为每栋建筑登记 BuildingSignature；完全相同签名为硬失败，换色/旋转/移动小装饰不算变体。
- 生活服务链：<食物、水、货运、住宿、生产、军防、文化娱乐与后勤如何跨区连接>；禁止用随机杂物冒充生活气息。
- 创新地标：<高大酒楼、画舫、水上会馆/戏台、桥市、临河仓楼等>；先做独立原型并验证结构、楼层、内部、码头/登船与夜景。
- 性能：builder 使用记录式 `StructurePlan`，AI 仍以全局坐标调用 `set_block` / `set_blocks` / `fill`；禁止创建完整密集 NumPy 画布、手工裁剪输出分片或 Python 三重循环遍历全城。体素审计通过 `render_region()` 生成扇区/建筑局部窗口。

风格与材料：
- <建筑风格>
- <主材、次材、强调材>
- 禁止：<不想出现的材料或形态>

知识库采用（有专项样板时必填）：
- 先读取：<知识目录/源文件>
- 必须采用的源函数或 prefab：<symbol 列表>
- 每条采用关系写入 src/<package>/reference_plan.py，包含 source、symbols、target、used_by、preserve、adapt、status。
- 必须迁移几何关系并由最终建筑真实调用；只提文件名、只写注释或复制未调用代码不算完成。
- work 必须自包含，不在运行时导入 knowledge/。

工程边界：
- 先读取 BRIEF.md 和 project.json。
- 只修改该 work 的 main.py 与 src/；不要手写 out/ 或网易配置。
- builder 返回 project.json 指定尺寸的一个逻辑画布；小型工程可用 `Structure`，预计密集索引达到 256 MiB 或任一水平轴不少于 1024 时必须用 `StructurePlan`。建筑源码不得感知 worldgen/ModSDK 分片。
- 最后必须运行 python scripts/structure_work.py build workset/<work_name>。
- 构建失败就继续修复源码，直到一次构建与审计通过。
- 若专项知识库有验收门槛，同时报告采用覆盖数和具体调用证据。
- 对可进入建筑同时报告大型建筑计划数、房间数、不可达房间、暗点、未登记光源和屋面/侧面漏点。
- 对室内同时报告 `portal_volume_blockers`、`partition_overwrites_portals`、`door_approach_blockers`、`entrance_route_breaks`、`rooms_disconnected_from_entrance`、`protected_route_intrusions`、`partition_ceiling_gaps`、`short_enclosed_partitions`、`partition_side_leaks`、`stair_swept_volume_blockers`、`stair_landing_blockers`、`stair_shaft_intrusions`、`physical_compartment_shortfall`、`unmaterialized_partitions`、`oversized_undivided_interiors`、`functionally_ambiguous_rooms`、`role_anchor_mismatches`、`unusable_furniture_groups` 和 `furniture_template_shortfall`。
- 对城防/高层建筑同时报告无支撑地板格、荷载路径失败、巡道连通分量、可达率、不可达门楼/角楼和浮空细部组件。
- 对含悬挂物、方向方块或景观的区域同时报告无锚链条、悬挂状态错误、错误朝向楼梯、楼梯路线断点、非法植物基底、种植区外散点、机械花阵和景观路线阻塞。
- 对超大型都市同时报告扇区完成/接缝、护城河与运河连续性、城区可达性、建筑签名冲突、换色伪变体、生活服务链缺口、创新地标原型和画舫登船关系。

完成时报告：源码模块、已实现区域、worldgen/ModSDK 切片数、维度 ID、推荐传送命令和仍需进游戏目测的项目。
```

如果尺寸、维度或世界坐标尚未确定，先运行：

```powershell
python scripts/structure_work.py new workset/<work_name> `
  --project-name "<display name>" `
  --size <X> <Y> <Z> `
  --origin <X> <Y> <Z>
```
